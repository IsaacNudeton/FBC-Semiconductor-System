Layer 2: Zynq Architecture

Two chips in one: ARM processors + FPGA fabric. How they talk to each other.

What is Zynq?

Zynq is Xilinx's SoC (System on Chip) that combines:

  • PS (Processing System) - Dual ARM Cortex-A9 cores + peripherals
  • PL (Programmable Logic) - The FPGA fabric we learned in Layer 1

Key Insight
The PS is a real computer - it boots, runs Linux (or bare-metal), has USB, Ethernet. The PL is custom hardware YOU design. They communicate via AXI buses.

Processing System (PS)

The PS boots first. It can then configure the PL with a bitstream.

Clock Generation

The PS generates clocks for both itself AND the PL:

PS_CLK (33.33 MHz input)
    │
    ├──► ARM PLL ──► 667 MHz (CPU clock)
    │
    ├──► DDR PLL ──► 533 MHz (memory clock)
    │
    └──► IO PLL ──► FCLK_CLK0 ──► 100 MHz ──► To PL (AXI bus clock)
                 └► FCLK_CLK1 ──► 200 MHz ──► To PL (vector timing)
                 └► FCLK_CLK2 ──► (unused)
                 └► FCLK_CLK3 ──► (unused)
Important for Our System
FCLK_CLK0 (100 MHz) is the AXI bus clock - all register accesses.
FCLK_CLK1 (200 MHz) is the base for vector timing clocks.

PS ↔ PL Communication

The PS and PL are connected by AXI (Advanced eXtensible Interface) buses.

┌─────────────────────────────────────────────────────────────────────┐ │ AXI Interconnect │ │ │ │ PS Side (Masters) PL Side (Slaves) │ │ ───────────────── ──────────────── │ │ │ │ ┌─────────────┐ AXI GP0 ┌──────────────────┐ │ │ │ ARM Core 0 │◄──────────────────►│ axi_fbc_ctrl │ │ │ │ │ (32-bit) │ (our registers) │ │ │ │ ARM Core 1 │ └──────────────────┘ │ │ └─────────────┘ AXI GP1 ┌──────────────────┐ │ │ ◄──────────────────►│ axi_io_table │ │ │ (32-bit) │ (pin config) │ │ │ └──────────────────┘ │ │ ┌─────────────┐ │ │ │ DMA │ AXI HP0 ┌──────────────────┐ │ │ │ Controller │◄══════════════════►│ Vector FIFO │ │ │ │ │ (64-bit, fast) │ (AXI-Stream) │ │ │ └─────────────┘ └──────────────────┘ │ │ │ │ ◄── = AXI4-Lite (registers, slow, simple) │ │ ◄══ = AXI4/AXI-Stream (data, fast, burst) │ └─────────────────────────────────────────────────────────────────────┘

AXI Port Types

PortWidthTypeUse
M_AXI_GP032-bitAXI4-LitePS reads/writes PL registers
M_AXI_GP132-bitAXI4-LiteMore PL registers
S_AXI_HP0-364-bitAXI4PL accesses DDR (DMA)
S_AXI_ACP64-bitAXI4Cache-coherent DDR access
Our Data Flow
  1. Firmware loads vectors into DDR memory
  2. Firmware configures DMA: "read from DDR address X, send to PL"
  3. DMA streams data through AXI HP0 → PL FIFO
  4. Vector engine pulls from FIFO, drives pins

2016 vs 2026: How We Use the PS

2016 (Sonoma with Linux)

Boot: FSBL → U-Boot → Linux
Time: ~30 seconds

Software Stack:
├── Linux kernel
├── Device drivers (AXI, DMA)
├── TCP/IP stack
├── Shell scripts
└── netcat for control

Problems:
- /tmp fills up → crash
- eth0 needs restart
- Complex, many failure points
- 30s boot per power cycle

2026 (FBC with Bare-metal Rust)

Boot: FSBL → Our firmware
Time: <1 second

Software Stack:
├── Bare-metal Rust
├── Direct register access
├── Minimal TCP (smoltcp)
└── Custom protocol

Benefits:
- No filesystem to corrupt
- No OS overhead
- Deterministic timing
- Fast boot, reliable

Memory Map

The ARM sees everything through memory addresses:

0x00000000 - 0x3FFFFFFF : DDR Memory (up to 1GB)
0x40000000 - 0x7FFFFFFF : PL AXI Slaves (M_AXI_GP0)
    0x43C00000 : axi_fbc_ctrl registers (our design)
    0x43C10000 : axi_io_table registers
    0x43C20000 : axi_vector_status
0x80000000 - 0xBFFFFFFF : PL AXI Slaves (M_AXI_GP1)
0xE0000000 - 0xE02FFFFF : PS Peripherals (UART, SPI, etc)
0xF8000000 - 0xF8FFFFFF : PS System registers (SLCR, etc)
0xFFFC0000 - 0xFFFFFFFF : On-chip RAM (256KB)
Register Access Example
// In bare-metal Rust:
const FBC_CTRL_BASE: u32 = 0x43C0_0000;
const FBC_STATUS_REG: u32 = FBC_CTRL_BASE + 0x00;
const FBC_CONTROL_REG: u32 = FBC_CTRL_BASE + 0x04;

// Read status
let status = unsafe { core::ptr::read_volatile(FBC_STATUS_REG as *const u32) };

// Write control
unsafe { core::ptr::write_volatile(FBC_CONTROL_REG as *mut u32, 0x01) };

QUESTIONS_AI_SHOULD_ANSWER:
  Q: How does the ARM write to FPGA registers?
  A: Memory-mapped I/O via AXI GP ports. Write to address 0x43C00000
     triggers AXI transaction to PL slave at that address.

  Q: How do vectors get from DDR to the FPGA?
  A: DMA controller reads DDR, streams via AXI HP port to PL FIFO.
     No CPU involvement during transfer.

  Q: Why bare-metal instead of Linux?
  A: Speed (1s vs 30s boot), reliability (no filesystem corruption),
     determinism (no scheduler), simplicity (fewer failure modes).
