Crystallization-based temperature control with pattern-aware feedforward

Why Not PID?

Traditional PID controllers require tuning: Kp, Ki, Kd values that must be calibrated per device, per environment. With 500+ boards, that's 500+ tuning sessions.

The Sonoma system uses bang-bang control (on/off) which is even worse - causing overshoot, undershoot, and oscillation around the setpoint.

Bang-Bang (Sonoma)

  • Simple on/off switching
  • Overshoot guaranteed
  • Oscillation around setpoint
  • No pattern awareness
  • Reactive only

ONETWO Crystallization (FBC)

  • Smooth convergence guaranteed
  • No overshoot possible
  • Constants from structure, not tuning
  • Pattern-aware feedforward
  • Predictive + reactive

ONETWO Framework Applied

The ONETWO framework defines a universal crystallization dynamic that applies to any settling system. The constants aren't arbitrary - they're structurally forced.

Settling Rate
e - 2 ≈ 0.718
Lock Iterations
7
Jitter Floor
10%

SETTLE = (e - 2): This is the rate at which error reduces per iteration. Not tunable - forced by the structure of crystallization dynamics.

LOCK = 7 iterations: After 7 iterations, the system has "crystallized" to within the floor. (0.718^7 ≈ 0.10)

FLOOR = 10%: Irreducible jitter. You can't do better than 10% of initial error due to measurement noise and physical limits.

The Control Algorithm

The controller follows the QUADRA pattern: Distinguish → Relate → Change → Embed

L1: Distinguish
error = setpoint - actual
→
L2: Relate
correction = error × 0.718
→
L3: Change
output += correction + ff
→
L4: Embed
check crystallization 
impl Thermal {
    pub fn update(&mut self, actual_mc: i32) -> ThermalOutput {
        // L1: Distinguish - what's the error?
        let error = self.setpoint_mc - actual_mc;

        // First iteration: capture initial error for floor calc
        if self.iterations == 0 {
            self.initial_error = abs(error);
        }

        // L2: Relate - calculate correction using settling rate
        // correction = error × (e-2) = error × 0.718
        let correction = (error * 718) / 1000;  // Fixed-point

        // L3: Change - accumulate output + feedforward
        self.output = self.output.saturating_add(correction);
        let output_with_ff = self.output.saturating_add(self.feedforward);
        let clamped = output_with_ff.clamp(-1000, 1000);

        // L4: Embed - check crystallization
        let error_abs = abs(error);
        let floor = (self.initial_error * 10) / 100;  // 10% floor
        let locked = error_abs <= floor.max(100);      // At least 0.1°C

        if self.iterations < 7 {
            self.iterations += 1;
        }

        ThermalOutput {
            correction: clamped,
            locked,
            error_mc: error,
            iterations: self.iterations,
        }
    }
}

Pattern-Aware Feedforward

Customer data showed that certain test patterns (like MBIST) cause massive current spikes - the Cisco 9402 data showed current jumping from 4.85A to 65.88A (13x increase) when switching patterns.

The thermal controller now analyzes vector toggle rate before executing to predict power and pre-compensate:

Physics: P = C × V² × f × α

  • C = capacitance (fixed by chip design)
  • V = voltage (fixed by power supply)
  • f = frequency (fixed by clock)
  • α = activity factor = toggle rate!

More toggles → more switching power → more heat. We can measure toggles from vectors.

/// Analyze vectors to estimate power draw
pub fn estimate_power(vectors: &[u128]) -> PowerEstimate {
    let mut total_toggles: u64 = 0;
    let mut total_active: u64 = 0;

    for i in 0..vectors.len() {
        // Active pins in this vector
        total_active += vectors[i].count_ones() as u64;

        // Toggles from previous vector (XOR)
        if i > 0 {
            let transitions = vectors[i] ^ vectors[i - 1];
            total_toggles += transitions.count_ones() as u64;
        }
    }

    let toggle_rate = total_toggles / (vectors.len() - 1);
    let active_pins = total_active / vectors.len();

    // Determine power level from physics
    let level = match (toggle_rate, active_pins) {
        (t, a) if t > 40 && a > 60 => PowerLevel::High,
        (t, a) if t > 20 || a > 40 => PowerLevel::Medium,
        _ => PowerLevel::Low,
    };

    PowerEstimate { level, toggle_rate, active_pins, vector_count }
}

Usage in Firmware

use hal::thermal::{Thermal, estimate_power};

// Create controller
let mut thermal = Thermal::new();
thermal.set_target(85_000);  // 85°C in milliCelsius

// When loading vectors, estimate power
let estimate = estimate_power(&vectors);
thermal.set_power_estimate(&estimate);

// Main loop
loop {
    let temp_mc = xadc.read_temperature_mc();
    let output = thermal.update(temp_mc);

    // Apply to hardware
    let heater_duty = output_to_heater(output.correction);
    let fan_duty = output_to_fan(output.correction);

    pwm.set_heater(heater_duty);
    pwm.set_fan(fan_duty);

    if output.locked {
        // Temperature stable, can proceed with test
    }
}
