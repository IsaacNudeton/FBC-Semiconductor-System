Layer 10: Bitstream & ONETWO

How we learned the FPGA's configuration format without vendor documentation

The Problem: Vendor Lock-in

Simple: Xilinx doesn't tell you how their bitstream format works. You need their tool (Vivado) to generate bitstreams. We wanted independence.
Traditional approach: Your design ──► Vivado ──► Bitstream ──► FPGA (black box) You can't see what Vivado does internally. You MUST use Vivado. It's slow, expensive, vendor-locked. What we wanted: Your design ──► OUR Toolchain ──► Bitstream ──► FPGA (we control it)
The challenge: A bitstream is millions of bits. Which bit controls which LUT? Which bit connects which wire? Xilinx doesn't publish this.

The Solution: ONETWO Pattern Learning

Simple: We looked at REAL bitstreams from Vivado and found the patterns. Like learning a language by listening, not reading a textbook.
ONETWO Method: 1. Get examples 2. Compare them 3. Find patterns Bitstream A A vs B "LUT data is always Bitstream B A vs C in words 4-11 for Bitstream C B vs C the lower slice" ┌───────────┐ ┌───────────┐ ┌───────────┐ │ ████░░░░█ │ │ Same here │ │ PATTERN: │ │ ░░░█████░ │ ───► │ Diff here │ ───► │ Word 4-5 │ │ █░█░█░█░█ │ │ Same here │ │ = LUT A │ └───────────┘ └───────────┘ └───────────┘
Key insight: What stays the same = structure. What changes predictably = configuration. What's random = noise (ignore it).
Technical: We used the `fbc-synth analyze` and `fbc-synth patterns` commands to extract recurring bit positions from the Vivado reference bitstream (kzhang_v2_2016/top.bit).

What We Discovered

Simple: Each "frame" (chunk of configuration) has 101 words. The first half (0-49) configures one slice, word 50 handles routing, the second half (51-100) configures another slice.
Frame Structure (101 words × 32 bits = 3232 bits per frame): ┌─────────────────────────────────────────────────────────┐ │ Words 0-49: CLB0 (lower slice) │ │ Words 4-5: LUT A (64 bits) │ │ Words 6-7: LUT B (64 bits) │ │ Words 8-9: LUT C (64 bits) │ │ Words 10-11: LUT D (64 bits) │ ├─────────────────────────────────────────────────────────┤ │ Word 50: Routing muxes + FF init bits │ ├─────────────────────────────────────────────────────────┤ │ Words 51-100: CLB1 (upper slice) │ │ Words 54-55: LUT A (64 bits) │ │ Words 56-57: LUT B (64 bits) │ │ Words 58-59: LUT C (64 bits) │ │ Words 60-61: LUT D (64 bits) │ └─────────────────────────────────────────────────────────┘
Component Location How We Found It
LUT A (lower slice) Words 4-5 64-bit clusters in pattern analysis
LUT B (lower slice) Words 6-7 Word spacing of 2
FF init values Word 50, bits 0-15 Most frequently set bits
LUT A (upper slice) Words 54-55 Same pattern, offset by 50
Routing muxes Word 50 54,000 bits set in Vivado (heaviest)

Verification: Our Output vs Vivado

Simple: We compared our bitstream patterns to Vivado's. The word pairs match exactly - our LUT data goes to the same places.

Result: LUT/FF configuration is correct. The remaining work is routing PIPs (the bits in word 50 that connect wires together).

The Complete Flow

Your Verilog Synthesis Technology Mapping ───────────── ────────► ─────────── ────► ────────────────── module adder( Gate-level LUT-level input a, b, netlist netlist output sum (AND, OR, XOR) (LUT6, FDRE) ); assign sum = a ^ b; ↓ ↓ ↓ ↓ Placement Routing ────────── ────────► ─────── Assign each Connect LUTs LUT to a using wires grid position ↓ ↓ Bitstream Gen ───────────── Write bits to configure each LUT, FF, wire ↓ .bit file
Commands:
fbc-synth build design.v -o output.bit - Full flow
fbc-synth analyze file.bit - See frame/bit stats
fbc-synth patterns file.bit - See word usage
fbc-synth diff a.bit b.bit - Compare two bitstreams
