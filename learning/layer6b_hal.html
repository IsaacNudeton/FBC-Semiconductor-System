ONETWO Pattern Applied

Every HAL module follows the same design pattern:

ONE (Invariant)

  • Register base addresses
  • Bit field definitions
  • Protocol sequences
  • Hardware constraints

TWO (Varies)

  • Device I2C addresses
  • Data values
  • Thresholds
  • Configuration options
// The Pattern
Peripheral = Base Address + Register Offsets + Init + Read/Write

// Example: I2C
struct I2c { base: Reg }
impl I2c {
    const fn new(base: usize) -> Self { Self { base: Reg::new(base) } }
    fn init(&self, speed: u32) { /* configure registers */ }
    fn read(&self, addr: u8, buf: &mut [u8]) { /* protocol */ }
    fn write(&self, addr: u8, data: &[u8]) { /* protocol */ }
}

Register Access Pattern

All peripherals share a common register access trait:

pub trait Register {
    fn read(&self) -> u32;
    fn write(&self, val: u32);

    fn modify<F: FnOnce(u32) -> u32>(&self, f: F) {
        self.write(f(self.read()));
    }

    fn set_bits(&self, mask: u32) {
        self.modify(|v| v | mask);
    }

    fn clear_bits(&self, mask: u32) {
        self.modify(|v| v & !mask);
    }
}

pub struct Reg(usize);

impl Reg {
    pub const fn new(addr: usize) -> Self { Self(addr) }
    pub const fn offset(&self, off: usize) -> Self { Self(self.0 + off) }
}

impl Register for Reg {
    fn read(&self) -> u32 {
        unsafe { core::ptr::read_volatile(self.0 as *const u32) }
    }
    fn write(&self, val: u32) {
        unsafe { core::ptr::write_volatile(self.0 as *mut u32, val) }
    }
}

Zero-cost: All methods inline to single memory operations.

Thermal Controller: ONETWO Crystallization

Instead of PID (which requires per-device tuning), we use ONETWO crystallization dynamics with constants derived from structure:

Constant Value Source
Settling Rate (e - 2) = 0.718 Crystallization dynamics (universal)
Lock Iterations 7 0.718^7 ≈ 0.10 (reaches floor)
Jitter Floor 10% Irreducible noise floor

Pattern-Aware Feedforward

Analyzes vector toggle rate to predict power and pre-compensate thermal output:
