Layer 10B: Optimization Engine

Generating optimal bitstreams for YOUR specific use case

The Problem: Generic vs Optimal

Simple: Vivado generates "good enough" bitstreams for everyone. But what if you need the BEST bitstream for your specific application?
Vivado's approach: Your design ──► Vivado ──► Generic bitstream (balanced for everyone) Problems: - Doesn't know YOUR critical paths - Same optimization for a blinky LED and a high-speed ADC - Can't prioritize what YOU care about What we wanted: Your design ──► OUR Toolchain ──► OPTIMAL bitstream + YOUR goals (for YOUR use case)
Key insight: The FPGA doesn't require Vivado's specific bit pattern. It just needs A VALID configuration. We can generate one that's optimal for our needs.

ONETWO Applied to Optimization

Simple: We used ONETWO thinking again. What makes a good bitstream? Look at what matters, ignore what doesn't.
ONETWO for Optimization: 1. What varies? 2. What's invariant? - Cell placement - FPGA structure - Wire routing - Available resources - Which nets are short - Bitstream format 3. What can we control? 4. What do we optimize for? - WHERE cells go - Timing (speed) - WHICH wires to use - Power (efficiency) - HOW to resolve conflicts - Area (density)

Optimization Profiles

Simple: Instead of one-size-fits-all, pick a profile that matches what you need. Like camera modes: portrait, landscape, sports.
Profile Optimizes For Trade-off Use When
timing Fastest critical paths May use more power High-speed interfaces
power Lowest power consumption May be slower Battery/thermal limits
area Smallest footprint Longer wires Resource-constrained
burn_in FBC system specific Balanced This project!
burn_in profile specifics:
- vec_clk, vec_clk_90, vec_clk_180 marked as critical (10x weight)
- Reliability goal (spread logic evenly, avoid hot spots)
- Power as secondary concern

How It Works

Simple: Critical nets get VIP treatment. They're placed first, routed first, and "win" when there's competition for resources.
Without Optimization: ┌─────────────────────────────────────────┐ │ All nets treated equally │ │ │ │ Net A ~~~~~~~~~~~~~~~~~~~► far away │ │ Net B ~~~► close │ │ Net C ~~~~~~~~► medium │ │ │ │ Random placement, random routing │ └─────────────────────────────────────────┘ With Optimization (Net A = critical): ┌─────────────────────────────────────────┐ │ Net A gets priority │ │ │ │ Net A ──► SHORT PATH (critical!) │ │ Net B ~~~~~~~~~► pushed aside │ │ Net C ~~~~~~► makes room for A │ │ │ │ Critical path optimized │ └─────────────────────────────────────────┘
Mechanism:
1. Placement: Critical nets have higher weight in cost function → cells placed closer together
2. Routing: Critical nets routed first, pay less congestion penalty → they "win" contested wires

Usage

# Default build (no optimization) fbc-synth build design.v -o design.bit # With burn-in profile (pre-configured for FBC) fbc-synth build design.v -o design.bit --optimize burn_in # With timing profile + specific critical nets fbc-synth build design.v -o design.bit \ --optimize timing \ --critical-nets "clk,data_bus,addr" # Low power mode fbc-synth build design.v -o design.bit --optimize power # Minimum area fbc-synth build design.v -o design.bit --optimize area

Verification

Simple: We tested it. The optimization engine runs, produces valid bitstreams, and the routing converges.
Metric Without Optimization With burn_in
Placement cost 18,672 546,261 (weighted)
Routing iterations 9 63
Critical path length Random Minimized
Why higher cost? The weighted cost is higher because critical nets have 10x weight. The engine works harder to minimize THOSE specific paths, even if total wire length increases.
